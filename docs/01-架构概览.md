## 1.1 架构类型

**模块化单体架构**

**选择理由**:

- 9人团队不需要微服务的运维复杂度
- MVP 阶段优先快速迭代和灵活调整
- 通过清晰的模块边界为未来拆分预留空间
- 单一部署单元降低 DevOps 成本

## 1.4 数据流向说明

### 用户发布故事流程

```
用户上传图片
    ↓
前端调用 /stories/upload-token 获取 OSS 临时凭证
    ↓
前端直传图片到 OSS (不经过后端)
    ↓
前端收到 OSS 返回的 URL
    ↓
前端调用 POST /stories 提交故事 (包含 OSS URL + 位置 + 内容)
    ↓
后端保存到 PostgreSQL (location 字段使用 PostGIS Point 类型)
    ↓
返回成功,前端跳转到地图查看
```

### 地图范围查询流程

```
用户拖动地图
    ↓
前端获取当前视野范围 (bounds)
    ↓
调用 GET /map/explore?lat=xxx&lng=xxx&radius=xxx
    ↓
后端使用 PostGIS ST_DWithin 查询范围内的点
    ↓
如果点数 > 100,执行聚合算法返回聚合数据
    ↓
前端渲染标点 (呼吸灯效果 + 情绪颜色)
```

## 1.5 核心技术亮点

| 技术点                   | 实现方式                   | 价值                    |
| ------------------------ | -------------------------- | ----------------------- |
| **自定义地图皮肤** | 高德 API `mapStyle` 配置 | 去除商业 POI,治愈系视觉 |
| **空间索引优化**   | PostGIS GIST 索引          | 10万点查询 < 200ms      |
| **点聚合算法**     | Supercluster.js (前端)     | 密集区域性能优化        |
| **OSS 直传**       | 后端签名 + 前端上传        | 减少服务器带宽压力      |
| **Shadowban 机制** | 数据库 visibility 字段     | 优雅隔离违规内容        |

## 1.6 系统边界说明

### 系统内 (我们开发)

- 用户认证 (邮箱  or  GitHub OAuth)
- 故事 CRUD + 地理位置查询
- 随机漫步算法
- 时光胶囊定时解锁
- 管理员内容审核

### 系统外 (第三方服务)

- 地图服务 (高德 Web API)
- 图片存储 (阿里云 OSS)
- 内容安全 (阿里云/腾讯云,可选)
- OAuth 认证 (GitHub)
