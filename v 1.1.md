## 🎨 一、 前端 UI/UX 体验改造 (重点要求)

从截图来看，目前的界面过于空旷，按钮细碎且缺乏呼吸感。请前端同学按以下规范进行重构：

### 1. 废弃全屏跳转，全面采用“悬浮弹窗 + 背景高斯模糊”
*   **痛点**：目前的发布页是一个全屏的白底页面，用户填表时完全脱离了地图场景，极其枯燥。
*   **优化方案**：
    *   **发布故事**、**查看个人信息/修改资料**、**消息列表**，这三个核心模块**全部改为在当前地图页上方弹出的 Dialog（居中弹窗）或 Bottom Sheet（底部抽屉）**。
    *   **视觉要求**：弹窗周围的遮罩层（Overlay）**不要用死板的纯黑色**，必须使用带有透明度的高斯模糊效果（CSS 建议：`backdrop-filter: blur(8px); background-color: rgba(255, 255, 255, 0.4)`）。
    *   **效果预期**：用户在填写故事时，依然能透过半透明的背景隐约看到地图，保持极强的“时空沉浸感”。

### 2. 悬浮按钮组（Dock 栏）整合
*   **痛点**：目前的随机漫步（骰子）在左下角，发布按钮（+号）在右下角，图标太小，显得地图边缘很碎。
*   **优化方案**：将核心操作按钮聚拢，做成一个悬浮的“胶囊型操作台（Dock）”，固定在屏幕正下方或右下角。
    *   **排版建议**：在一个半透明的胶囊底板上，依次排列：`[ 👤 我的 ]`  `[ 🎲 随机漫步 ]`  `[ ➕ 发布故事 (高亮/放大) ]`  `[ 🔔 消息 ]`。
    *   这样不仅美观现代，而且用户的视觉焦点更集中。

### 3. 右侧“附近的故事”面板优化
*   **痛点**：目前右侧的白色侧边栏一直死死占用屏幕空间，导致地图视野变小。
*   **优化方案**：改为**抽屉式（Drawer）**或**悬浮卡片式**。默认隐藏，用户点击屏幕上的某个聚合点，或点击“查看附近”按钮时，才从侧边滑出或从底部弹起。

## ⚙️ 二、 后端 API 基础业务闭环补充

之前的 API 忽略了最基本的用户体验闭环。用户连修改昵称的地方都没有，且发了内容不知道谁评论了。请后端补充以下基础接口：

### 1. 个人资料与账号设置 (User Profile & Settings)
*连改名字和头像都不行的话，产品的用户留存率会大打折扣。*
*   **`GET /api/users/:id`** - **[新增]** 查看其他用户的公开主页（仅返回该用户公开的帖子列表、昵称、头像，隐藏私密帖子）。
*   **`PUT /api/users/me`** - **[新增]** 更新个人资料（支持修改 `nickname` 昵称, `avatar` 头像 URL, `bio` 个性签名）。
*   **`PUT /api/users/me/password`** - **[新增]** 修改密码（提供原密码和新密码）。
*   **`DELETE /api/users/me`** - **[新增] 注销账号**。

### 2. 内容状态管理与纠错 (Content Management)
*用户发文肯定会有错别字，或者发完后悔想隐藏，我们必须给他们后悔药。*
*   **`PUT /api/stories/:id`** - **[新增]** 编辑故事内容（仅限作者本人，仅允许修改文字 `content` 和情绪标签 `mood_tag`，地理位置不可更改）。
*   **`PUT /api/stories/:id/visibility`** - **[新增]** 切换可见性。比如一键把某条公开帖子转为“仅自己可见”（设为私密），或者关闭匿名状态。

### 3. 收藏夹机制 (Bookmark / Collection)
*“抱抱/点赞”是情绪表达，但“收藏”是高价值内容的沉淀。用户在随机漫步时看到极美的风景照，需要有个地方存下来。*
*   **`POST /api/collections/:story_id`** - **[新增]** 收藏某条故事。
*   **`DELETE /api/collections/:story_id`** - **[新增]** 取消收藏。
*   **`GET /api/stories/me/collections`** - **[新增]** 获取“我的收藏”列表（前端在个人中心展示）。

### 4. 消息中心与通知闭环 (Message & Notification)
*如果没有通知，用户发完帖子就走了，不会再打开 App。消息中心是唤醒用户的核心。*
*   **`GET /api/notifications`** - **[新增]** 获取当前用户的消息列表。
    *   *数据结构需包含*：`type` (枚举：系统公告、收到点赞、收到评论)、`sender_info` (谁操作的)、`story_id` (关联帖子的跳转ID)、`is_read` (是否已读)。
*   **`PUT /api/notifications/:id/read`** - **[新增]** 标记单条/全部消息为已读。

### 5. 地理逆解析与全局搜索 (LBS & Search)
*   **`GET /api/map/geocode`** - **[新增]** 坐标转文字地址。
    *   *业务说明*：前端拿到经纬度后，传给后端，后端调用高德/腾讯地图的逆地理编码服务，返回给前端“北京市海淀区中关村”这样的文字。**这样发布的帖子上就能带上具体的文字地名，极其增加真实感。**
*   **`GET /api/search/stories`** - **[新增]** 关键词全站搜索。
    *   *业务说明*：支持模糊搜索帖子的文字内容，方便用户查找特定的故事（如搜关键词“考研”、“看海”）。

### 6. 用户反馈与求助 (Feedback)
*   **`POST /api/feedback`** - **[新增]** 用户提交意见反馈或 Bug 报告（传 `content` 和可选的 `images`）。
    *   *业务说明*：提交后，管理员可以在后台看到。这同样是一个展现项目“闭环完整度”的极佳功能。

### 7. 评论与回复功能的微调 (Comments)
为了让用户能产生交流，评论接口需要微调：
*   **`POST /api/stories/:id/comments`** - 发布评论。
    *   **[字段要求]**：Payload 中支持传入可选参数 `reply_to_comment_id`。
    *   *业务逻辑*：我们不需要做复杂的楼中楼 UI，但用户评论时，如果带着这个 ID，说明他是在回复某个人。后端只需把这条回复与原帖子关联，并给被回复的人触发一条 Notification 即可。前端展示时，可以直接显示为 “小A 回复 小B：今天天气真好”。
*   **`GET /api/stories/:id/comments`** - 获取故事评论列表（按时间排序即可）。

## 🛡️ 三、 管理员后台与社区治理闭环 (Admin & Moderation)
*PM 业务说明：作为一个主打“治愈”的 UGC（用户生产内容）社区，如果没有完善的审核和治理机制，产品上线三天就会被广告和低俗内容淹没。因此，后端必须提供以下配套的 Admin API。*

### 1. 核心数据看板 (Dashboard Stats)
*   **`GET /api/admin/dashboard/stats`** - 获取大盘数据。
    *   *返回内容*：今日新增用户数、今日新增故事数、待处理的举报数量。
    *   *作用*：让管理员一登录后台就能看到社区活跃度和工作量。

### 2. 内容策展与隐形守护 (Content Curation & Shadowban)
*   **`POST /api/admin/stories/:id/recommend`** - 设为/取消“精选推荐”。
    *   *业务逻辑*：被标记为推荐的故事，在前端地图缩放或请求列表时，拥有更高的展示优先级（也可以单独在前端做一个“精选流”展示）。
*   **`POST /api/admin/stories/:id/shadowban`** - 隐形降权（仅作者可见）。
    *   *业务逻辑*：对应我们在产品宣讲时提到的“守夜人”机制。对于轻微违规或发小广告的帖子，不直接删除（防止对方换号重发），而是将其状态改为 `is_shadowbanned = true`。此后，除了发布者本人，全网其他用户请求地图 API 时均过滤掉该帖子。

### 3. 举报处理全链路 (Report Handling)
*   **`POST /api/reports`** - **[C端接口]** 用户提交举报（关联 `story_id`、`user_id` 和 `reason`）。
*   **`GET /api/admin/reports`** - **[B端接口]** 管理员分页获取被举报的内容列表，按时间倒序。
*   **`POST /api/admin/reports/:id/handle`** - **[B端接口]** 管理员处理该举报。
    *   *Payload 传参建议*：`action_type` (枚举值：`ignore` 忽略, `delete_story` 直接删帖, `shadowban` 降权隐藏, `ban_user` 直接封禁发布者)。
    *   *业务逻辑*：后端根据 `action_type` 自动执行对应操作，并将该条举报记录标记为“已处理”。

### 4. 账号封禁机制 (User Banning)
*   **`POST /api/admin/users/:id/ban`** - 封禁/解封用户账号。
    *   *业务逻辑*：修改 user 表的 `status` 为 `banned`。该用户再次请求鉴权接口（或任何需要 token 的接口）时，后端统一返回 `403 Forbidden` 及提示信息“您的账号因违规已被封禁”。

## 📝 PM 总结与排期建议
这次的 V1.1 迭代没有增加任何复杂的算法（之前的时光胶囊等逻辑如果还没做可以往后稍稍）。

1. **前端同学**：重点把全屏表单改成带有毛玻璃效果的**弹窗（Modal）**，把散落的按钮**聚合成一个底部控制台（Dock）**，

2. **后端同学**：把建表时漏掉的**消息通知表（Notifications）**加上，确保“发布 -> 被评论 -> 收到消息通知 -> 修改资料”这个最基础的活人使用链路能够跑通。
